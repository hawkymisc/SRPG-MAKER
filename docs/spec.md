# シミュレーションRPGツクール 要件定義・仕様書

**版数:** v0.2
**作成日:** 2026-06-11
**開発前提:** 実装はAIエージェントに全面委任 / 配布はPC(Win・mac・Linux)+Web+モバイル(**コンソールは対象外と決定**)

---

## 1. プロダクト概要

### 1.1 目的

プログラミング知識がなくても、ファイアーエムブレム/タクティクスオウガ系の「グリッドベース・ターン制シミュレーションRPG」を制作・配布できるツールを提供する。エディタ(制作環境)とランタイム(ゲーム実行エンジン)の2層で構成する。

### 1.2 想定ユーザー

| ペルソナ | 特徴 | 主なニーズ |
|---|---|---|
| 趣味の個人開発者 | プログラミング経験なし〜初級 | GUIだけで完結、サンプル素材、すぐ動く |
| 同人ゲーム制作者 | ツクール系経験者 | カスタマイズ性、独自素材の取り込み、配布 |
| 上級者・拡張開発者 | JS/TSが書ける | プラグインAPI、スクリプトによる戦闘式の改変 |

### 1.3 競合・差別化

SRPG Studio(Windows専用)、RPGツクール(SRPG非対応)、Unityテンプレート(学習コスト高)に対し、**「ブラウザで動くエディタ+マルチプラットフォーム出力+モダンなJSプラグインAPI」**で差別化する。

## 2. スコープ定義

### 2.1 MVP(フェーズ1)

- マップエディタ(タイルベース、地形効果設定)
- ユニット/クラス/武器/アイテム/スキルのデータベースエディタ
- 戦闘の基本ループ(自軍→敵軍のターン制、移動・攻撃・待機)
- 勝利/敗北条件(敵全滅・ボス撃破・ターン数・拠点防衛)
- シンプルなイベントシステム(開戦時・ユニット撃破時・ターン開始時など)
- 簡易敵AI(突撃・待ち伏せ・防衛・移動のみ)
- テストプレイ機能(エディタ内で即実行)
- HTML5書き出し(ブラウザで遊べる成果物)

### 2.2 フェーズ2

- 章(シナリオ)管理、章間の拠点画面(ユニット編成・ショップ・会話)
- 支援会話・成長率・クラスチェンジ
- ビジュアルイベントエディタ、会話シーン演出
- プラグインAPI公開、戦闘計算式のカスタマイズ
- デスクトップ書き出し(Electron:Win/mac/Linux)

### 2.3 フェーズ3(将来)

- 戦闘アニメーション(専用バトル画面)
- モバイル書き出し(Capacitor:iOS/Android)、クラウド保存、素材ストア

### 2.4 スコープ外(確定)

- **コンソール展開(Switch等)** — Web技術スタックでは移植不能なため明示的に対象外とする
- 3Dマップ、リアルタイム戦闘、オンライン対戦、MMO要素

## 3. 技術選定

### 3.1 選定方針

前提が2つ確定している:**(a) 実装はAIエージェントに全面委任、(b) 配布はPC+Web+モバイル(コンソールなし)**。

この前提では選定基準が変わる。「人間が書く労力」より、**エージェントが正確に書けること(学習データ量)と、エージェントが自己検証できること(ヘッドレステスト可能性)**が支配的になる。両方で最強なのはTypeScript+Webエコシステムであり、全面採用する。Unityのようなエディタ画面のGUI操作を前提とする開発フローはエージェントと致命的に相性が悪く、Godotも検証ループの組みにくさで劣後する。

### 3.2 比較検討

| 候補 | エージェント開発適性 | マルチ出力 | 判定 |
|---|---|---|---|
| **TypeScript + Web技術** | ◎ 学習データ最大・ヘッドレス検証容易 | ◎ PC/Web/モバイル全対応 | ✅ 採用 |
| Unity (C#) | △ GUI操作中心の開発フロー、シーンファイルがバイナリ的 | ◎(コンソール含む) | ❌ |
| Godot | ○ シーンはテキストだが検証ループが弱い | ○ | ❌(コンソール断念により利点消失) |
| ネイティブ (C++/Rust) | △ 検証コスト大 | △ Web出力困難 | ❌ |

### 3.3 採用スタック(v0.1からの変更点は太字)

| レイヤー | 技術 | 選定理由 |
|---|---|---|
| 言語 | TypeScript (strict) | 型安全。エージェントのガードレールとしても機能 |
| ゲームエンジン | **Phaser 3**(v0.1のPixiJSから変更) | エージェントは画面を目視検証できないため、タイルマップ・カメラ・入力など視覚バグが出やすい部分は実績済みエンジンに委ね、自作を戦闘ロジックに集中させる |
| エディタUI | React 18 + Zustand | 複雑フォームUIに最適。Playwrightで自動E2E検証可能 |
| デスクトップ | **Electron**(v0.1のTauriから変更) | Chromium同梱により全OSで描画が同一挙動。「作ったゲームがどこでも同じに動く」信頼性を優先。エージェントの知識量も最大 |
| モバイル書き出し | **Capacitor**(フェーズ3) | Webランタイムをそのままラップ |
| 乱数 | **シード付き決定論的RNG(必須要件)** | 戦闘の完全リプレイ=ゴールデンマスターテストを可能にし、エージェントの自動回帰検証の基盤とする |
| データ定義 | Zod スキーマ | 型生成・フォーム生成・読込検証を一元化。エージェントの出力をスキーマで拘束 |
| 保存形式 | JSONファイル群(Git管理可能) | 差分管理・テキスト修正・エージェントによる直接生成が可能 |
| ビルド | Vite + pnpm workspaces | editor / runtime / shared / plugin-api の monorepo |
| テスト | Vitest(単体)+ Playwright(E2E)+ スクリーンショット回帰 | エージェントの自己検証ループの中核 |
| 音声 | Howler.js | クロスプラットフォーム |

### 3.4 リポジトリ構成

```
srpg-maker/
├── packages/
│   ├── shared/      # 型定義・Zodスキーマ・戦闘計算・AI思考(純粋関数のみ・描画依存禁止)
│   ├── runtime/     # ゲームエンジン(Phaser、シーン管理、セーブ)
│   ├── editor/      # エディタアプリ(React)
│   └── plugin-api/  # 公開プラグインAPI(フェーズ2)
├── assets-default/  # 同梱サンプル素材(CC0)
├── templates/       # 新規プロジェクトテンプレート
└── e2e/             # Playwrightシナリオ・ゴールデンマスター
```

## 4. AIエージェント開発要件(本版で新設)

実装をエージェントに委任するため、以下を**機能要件と同格の必須要件**とする。

### 4.1 検証戦略(エージェントの自己検証ループ)

| 層 | 手法 | 内容 |
|---|---|---|
| 戦闘ロジック | ゴールデンマスターテスト | シード固定の戦闘シナリオ(初期配置+入力列)→結果スナップショットを保存。ロジック変更時に全リプレイを自動照合 |
| データ整合性 | Zodスキーマ検証 | 全JSON読み書きでスキーマ検証必須。サンプルプロジェクトをCIで常時ロード検証 |
| エディタUI | Playwright E2E | 「マップ作成→ユニット配置→テストプレイ→クリア」の一気通貫シナリオを常時グリーンに保つ |
| 描画 | スクリーンショット回帰 | 主要画面のピクセル差分検証(閾値付き)。エージェントが目視できない領域を機械検証で代替 |
| 全体 | CIゲート | typecheck / lint / test / e2e の全パスをマージ条件にする。エージェントは赤いCIをマージ不可 |

### 4.2 エージェント向け設計ガードレール

1. **決定論の徹底** — RNGはシード注入のみ。`Date.now()` や `Math.random()` の直接使用をlintで禁止。
2. **純粋関数境界** — `shared` パッケージはDOM/Phaser/Node APIへの依存をlintで禁止。戦闘解決は `(state, action, rng) => newState` の形式に統一。
3. **スキーマ駆動** — データ構造の変更はZodスキーマの変更から始める(スキーマ→型→マイグレーション→実装の順序を規約化)。
4. **タスク分割粒度** — 1タスク=1パッケージ内で完結し、既存テストを壊さず新テストを伴う単位に分割する。マイルストーンはこの粒度まで分解してから委任する。
5. **アーキテクチャ決定記録(ADR)** — 設計判断は `docs/adr/` に記録し、エージェントのコンテキストとして常時参照させる。

## 5. 機能要件

### 5.1 エディタ機能

#### 5.1.1 プロジェクト管理
- 新規作成(テンプレート選択)/ 開く / 保存 / 自動バックアップ(世代管理5件)
- プロジェクト設定:解像度(既定 1280×720)、タイルサイズ(32/48/64px)、フォント、タイトル

#### 5.1.2 マップエディタ
- タイルパレットからのペイント(ペン・矩形・塗りつぶし)、レイヤー3層(下層/上層/オブジェクト)
- 地形タイプ定義:移動コスト(クラス別)、回避補正、防御補正、回復床、進入不可
- ユニット配置:敵・NPC・自軍初期配置・増援(出現ターン/条件付き)
- 出撃マス、離脱マス、宝箱・扉・村などのギミック配置
- マップサイズ:最小10×10〜最大100×100

#### 5.1.3 データベースエディタ
| タブ | 主な項目 |
|---|---|
| ユニット | 名前、顔/マップ画像、クラス、Lv、初期能力、成長率、所持品、スキル |
| クラス | 移動タイプ(歩兵/騎馬/飛行/重装)、移動力、武器適性、能力上限、上位クラス |
| 武器 | 種別(剣/槍/斧/弓/魔法…)、威力、命中、必殺、射程(min-max)、重さ、耐久、特効タグ |
| アイテム | 回復量、効果種別、使用回数、価格 |
| スキル | 発動条件、効果(計算式パラメータ参照可)、パッシブ/アクティブ |
| 地形 | 5.1.2の地形タイプ一覧 |
| 用語/システム | ダメージ計算式の選択、武器三すくみのON/OFF、経験値テーブル、乱数方式 |

#### 5.1.4 イベントエディタ
- トリガー:章開始/章終了/ターン開始(N)/ユニット撃破/指定マス到達/会話コマンド選択/宝箱開封
- コマンド(MVPは15種に固定):メッセージ表示、選択肢、ユニット出現/離脱、変数・スイッチ操作、条件分岐、BGM/SE、画面効果、勝敗条件の変更、章移動 ほか
- 実装形式:コマンドリスト方式。内部表現はJSONのコマンド配列

#### 5.1.5 テストプレイ
- ワンクリックで現在の章を起動
- デバッグ機能:ユニット無敵、全マス移動、変数ビューア、戦闘ログ出力、**シード固定再生(不具合再現用)**

### 5.2 ランタイム(ゲーム)機能

#### 5.2.1 戦闘フロー
1. 章開始イベント → 出撃準備(フェーズ2)→ 戦闘マップ開始
2. ターン制:自軍フェイズ → 敵軍フェイズ → 第三軍フェイズ
3. ユニット選択 → 移動範囲表示(ダイクストラ法・地形コスト考慮)→ 移動 → 行動選択(攻撃/スキル/アイテム/待機)→ 戦闘解決
4. 勝利条件達成で章クリア、敗北条件でゲームオーバー(リトライ選択)

#### 5.2.2 戦闘計算(既定式・エディタで係数変更可)
```
攻撃力   = 力(or 魔力) + 武器威力 + 特効補正 + 相性補正
ダメージ = 攻撃力 − (守備 or 魔防) − 地形防御補正   (最低0)
命中率   = (技×2 + 幸運/2 + 武器命中) − (回避側: 速さ×2 + 幸運 + 地形回避)
必殺率   = (技/2 + 武器必殺) − 相手の幸運
追撃     = 速さ差が閾値(既定4)以上で2回攻撃
```
乱数方式は「単乱数/2回乱数平均」を選択可。**すべての乱数消費はシード付きRNG経由とし、入力列から戦闘を完全再現できること。**

#### 5.2.3 敵AI(MVP)
| タイプ | 挙動 |
|---|---|
| 突撃 | 最も撃破期待値の高い対象へ移動して攻撃 |
| 待ち伏せ | 攻撃射程に入った敵のみ反応 |
| 防衛 | 指定マスから動かず射程内のみ攻撃 |
| 移動のみ | 指定地点へ移動(離脱・増援役) |

評価関数:`与ダメ期待値 × 命中率 + 撃破ボーナス − 被反撃期待値`。AI思考も `shared` の純粋関数とし、単体テスト対象とする。

#### 5.2.4 セーブ/ロード
- マップ中セーブ(中断)+ 章間セーブ(本セーブ3スロット以上)
- セーブデータはスキーマバージョン付きJSON(互換マイグレーション対応)

```
MyGame/
├── project.json        # プロジェクト設定・スキーマバージョン
├── database/
│   ├── units.json / classes.json / weapons.json
│   ├── items.json / skills.json / terrain.json

    ├── images/ (tilesets, units, faces, ui)
    └── audio/  (bgm, se)
```

### ユニット定義の例

```json
{
  "id": "unit_alm",
  "name": "アルム",
  "classId": "class_lord",
  "level": 1,
  "stats":  { "hp": 20, "str": 7, "mag": 0, "skl": 6, "spd": 7, "lck": 5, "def": 5, "res": 2, "mov": 5 },

```

全データは `shared` のZodスキーマで定義し、エディタのフォーム生成・読込検証・型生成を一元化する。

## 8. アーキテクチャ概要

```
┌─────────────── Editor (React) ───────────────┐
│ マップエディタ │ DBエディタ │ イベントエディタ │
│         └── Zustand Store(プロジェクト状態)   │
└───────────────┬──────────────────────────────┘
                │ JSON(プロジェクトデータ)
┌───────────────▼──────────────────────────────┐
│ Runtime (Phaser 3)                            │
│  SceneManager(タイトル/マップ/会話/メニュー)   │
│  BattleSystem ← shared/battle-logic(純粋関数) │
│  Input │ Audio │ SaveManager                  │
└──────────────────────────────────────────────┘
```

設計原則:
1. **ロジックと描画の分離** — 戦闘解決・移動範囲・AIは `(state, action, rng) => newState` の純粋関数。
2. **決定論** — シードと入力列があれば戦闘を完全再現できる(リプレイ・自動検証・不具合再現の基盤)。
3. **データ駆動** — ランタイムはJSONを解釈するだけで、ゲーム固有コードを持たない。
4. **イベントはコマンドのインタプリタ** — async generatorで会話待ち・選択待ちを表現。

## 10. リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| エージェント生成コードの品質ばらつき | バグ混入 | CIゲート必須化、ゴールデンマスター、スキーマ拘束、ADR参照 |
| 視覚的バグの見落とし(エージェントは目視不可) | 体験悪化 | Phaser採用で自作描画を最小化+スクリーンショット回帰 |
| エディタUIの開発規模肥大 | 工数超過 | Zodスキーマからのフォーム自動生成 |
| データスキーマの破壊的変更 | プロジェクト破損 | schemaVersion+マイグレーション関数を初期から導入 |
| イベント仕様の膨張 | 完成遅延 | MVPはコマンド15種に固定、要望はプラグインAPIへ |
| 素材権利問題 | 配布トラブル | 同梱素材はCC0/自社制作のみ |
